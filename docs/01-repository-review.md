## 2. 問題点の洗い出し

空リポジトリのため「既存コードの問題」はありませんが、プロジェクト立ち上げ観点で以下が未決定・リスクです。

### 2.1 未決定事項（ユーザー判断が必要）

| # | 項目 | 推奨 | 理由 |
|---|---|---|---|
| U-1 | **ライセンス** | 当面プライベート想定なら「All rights reserved」（LICENSEなし）。OSS化するなら AGPL-3.0 | レシピ生成アルゴリズムがコアIP。MIT だと商用コピーが容易 |
| U-2 | **リポジトリ名** | `coffee` → `coffee-recipe-lab` へのリネームを推奨 | 検索性・将来の複数リポジトリとの区別 |
| U-3 | **独自ドメイン** | 早期取得を推奨（例: `coffeerecipelab.app`） | Cloudflare 管理下に置くと Workers/認証設定が単純化 |
| U-4 | **Cloudflare 有料プラン** | MVP は Free で可。β以降 Workers Paid ($5/月) | D1 read/write 上限・Workers CPU 時間の余裕 |

### 2.2 技術的リスクと対策

| # | リスク | 影響 | 対策 |
|---|---|---|---|
| R-1 | **グラインダー目盛↔粒度(μm)の対応データの精度**。公式のμm公表値がないグラインダーが多い（De'Longhi KG521J-M 含む） | レシピの再現性が下がる | 対応表を「参考値+信頼度ラベル」として持ち、ユーザーキャリブレーション機能で補正（docs/11） |
| R-2 | **Web Bluetooth は iOS Safari 非対応** | 将来のスケール連携が iOS で不可 | v2.0 で検討。設計上は BLE 層を抽象化しておき、必要なら Capacitor ラッパーで対応（docs/02 §7） |
| R-3 | **D1 は単一リージョン SQLite**（10GB上限、書き込みは単一プライマリ） | 数万人規模での書き込み集中 | 読み取り中心の設計 + Sessions API / キャッシュ。将来の分割戦略を docs/07 に明記 |
| R-4 | **レシピの科学的根拠には限界がある**（官能評価は主観・豆の個体差大） | 「最適」の過信によるUX毀損 | 生成レシピに必ず根拠と調整余地を表示（説明可能性）。フィードバックループで個人最適化 |
| R-5 | **Next.js on Workers（OpenNext）のバンドルサイズ・互換性制約**（Node API 一部非対応） | ビルド失敗・遅延 | OpenNext は 2026年に 1.0 GA 済み。`nodejs_compat` フラグ + CI でサイズ監視（docs/13） |
| R-6 | 空リポジトリのため**ブランチ保護等が未設定** | main への直接push事故 | docs/04 の Git 運用ルールに従い GitHub 設定を初期セットアップ時に適用 |

## 3. 初期セットアップで必要な作業（実装フェーズの最初のPR）

1. `.gitignore` / `.editorconfig` / `.nvmrc`（本設計で追加済みのものを利用）
2. pnpm workspace の scaffolding（docs/03 の通り）
3. GitHub リポジトリ設定:
   - main ブランチ保護（PR必須・CI必須・force-push禁止）
   - squash merge のみ許可
   - Issue / PR テンプレート
4. CI（lint / typecheck / test / build）を最初に通す — **コードより先にCI**

## 4. 結論

既存資産ゼロのため、本設計書群（docs/02〜15）をそのまま初期設計として採用する。
上記 U-1〜U-4 は実装開始前にオーナー判断を仰ぐが、**どれも設計をブロックしない**
（ライセンス以外はデフォルト推奨で進めて後から変更可能）。
