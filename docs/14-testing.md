# 14. テスト戦略

## 1. 方針: テストピラミッド + エンジン最重視

```
        E2E (Playwright)        少数・中核ループのみ
      統合 (API + D1)           Hono ルート単位
    ユニット (Vitest)            エンジンは網羅的に
```

| 対象 | ツール | カバレッジ目標 |
|---|---|---|
| packages/engine | Vitest | **90%+（必須）** |
| packages/db | drizzle スキーマは型で担保 + マイグレーション適用テスト | — |
| apps/web hooks/lib | Vitest (happy-dom) | 70% |
| API ルート | Vitest + @cloudflare/vitest-pool-workers（miniflare 上で D1/KV 実体） | 主要パス100% |
| UI フロー | Playwright | 中核ループ1本 + 各画面スモーク |

## 2. エンジンのテスト（プロダクトの生命線）

### 2.1 ゴールデンテスト（`packages/engine/test/golden/`)
- 代表 30 入力（豆5 × 器具11 の主要組合せ + Iced + 極端値）の生成結果を JSON スナップショット保存
- 係数を変更する PR ではスナップショット差分が「レシピがどう変わるか」のレビュー材料になる
- スナップショット更新は `pnpm --filter engine test:golden -u` + PR 説明に変更理由必須

### 2.2 プロパティテスト（fast-check、devDependency は可）
不変条件（docs/10 §10）を任意入力で検証:
- 出力レンジ: 湯温 78–97 / 比率 1:10–1:20 / 粒度が器具レンジ内
- 構造: ステップ atSec 単調増加 / pour の累計湯量が単調増加し最終値 = waterG / valve open/closed の整合
- 決定論: 同一入力で完全同一出力（2回呼んで deepEqual）
- 単調性: strength を上げると ratio は締まる方向にのみ動く、等

### 2.3 官能検証（人力・自動化外）
- リリース前チェックリスト: 主要 10 レシピを実際に抽出して味を確認（オーナー + 将来はコミュニティβ）
- 結果は Issue テンプレート「官能QA」で記録し、係数調整の根拠として docs/10 に反映

## 3. API 統合テスト

- `@cloudflare/vitest-pool-workers` で Workers ランタイム上のテスト（本物の D1 SQLite / KV エミュレーション）
- 各ルート: 正常系 / バリデーション 400 / 認可 401・403 / レートリミット 429
- マイグレーションテスト: 空 DB に全マイグレーションを順適用 → スキーマが Drizzle 定義と一致すること

## 4. E2E（Playwright）

MVP でのシナリオ:
1. **中核ループ**: オンボーディング → ウィザード4ステップ → レシピ表示（数値が妥当レンジ）→ タイマー完走（fake timers）→ フィードバック → ログに記録
2. スモーク: 全画面が 200 + 主要要素表示（axe-core による a11y 違反ゼロもここで検証）
3. ゲスト永続化: リロードしてもデータが残る

- 実行: `wrangler dev`（本番同等ランタイム）に対して。CI では main + `e2e` ラベル PR のみ
- ビジュアルリグレッションは導入しない（メンテコスト > 価値。デザイントークン規約で代替）

## 5. テストを書く基準（実装者向け）

- バグ修正 PR には再現テスト必須
- エンジンの係数・ルール追加 = ゴールデン or プロパティテスト追加とセット
- UI の分岐ロジックは hooks に抽出してユニットテスト（コンポーネント自体のテストは最小限）
- モックは境界（Repository / fetch）のみ。内部実装のモック禁止
